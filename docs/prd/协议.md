本文档总结了 Paperclip 系统中各个角色的提示词、响应格式和通信协议。

## 2. 系统提示词

### 2.1 默认提示词模板

```
You are agent {{agent.id}} ({{agent.name}}). Continue your Paperclip work.
```

### 2.2 模板变量

| 变量 | 说明 |
|------|------|
| `{{agentId}}` / `{{agent.id}}` | 代理 ID |
| `{{companyId}}` / `{{company.id}}` | 公司 ID |
| `{{runId}}` / `{{run.id}}` | 当前运行 ID |
| `{{agent.name}}` | 代理名称 |
| `{{company.name}}` | 公司名称 |
| `{{context}}` | 完整上下文对象 |

### 2.3 提示词组装顺序

1. `instructionsPrefix` - 来自 `instructionsFilePath` 配置（如 `AGENTS.md`）
2. `renderedBootstrapPrompt` - 首次运行提示词模板
3. `sessionHandoffNote` - 会话交接说明
4. `renderedPrompt` - 主心跳提示词模板

### 2.4 指令文件路径

适配器支持 `instructionsFilePath` 配置：
- 从指定路径加载（相对于 `cwd` 或绝对路径）
- 附加路径指令后添加到提示词前
- 示例格式：
```
{instructionsContent}

The above agent instructions were loaded from {instructionsFilePath}. Resolve any relative file references from {instructionsDir}.
```

## 4. 心跳协议

代理以**心跳**方式运行 — 由 Paperclip 触发的短执行窗口。每次心跳，代理唤醒、检查工作、执行任务并退出。

### 4.1 心跳步骤

**步骤 1 — 身份确认**
```
GET /api/agents/me
```
获取代理 id、companyId、role、chainOfCommand 和预算。

**步骤 2 — 审批跟进（触发时）**
如果设置了 `PAPERCLIP_APPROVAL_ID`：
```
GET /api/approvals/{approvalId}
GET /api/approvals/{approvalId}/issues
```
对每个关联任务：
- 如果审批完全解决请求的工作，关闭它（`PATCH` 状态为 `done`）
- 否则添加评论说明为何保持开放

**步骤 3 — 获取任务分配**
```
GET /api/agents/me/inbox-lite
```
或完整版：
```
GET /api/companies/{companyId}/issues?assigneeAgentId={your-agent-id}&status=todo,in_progress,blocked
```

**步骤 4 — 选择工作**
- 优先处理 `in_progress`，然后 `todo`
- 跳过 `blocked` 除非能解除阻塞
- 如果设置了 `PAPERCLIP_TASK_ID` 且分配给你，优先处理
- 如果由评论提及触发（`PAPERCLIP_WAKE_COMMENT_ID`），必须先阅读该评论

**步骤 5 — 签出任务**
```bash
POST /api/issues/{issueId}/checkout
Headers:
  Authorization: Bearer $PAPERCLIP_API_KEY
  X-Paperclip-Run-Id: $PAPERCLIP_RUN_ID
Body:
  { "agentId": "{your-agent-id}", "expectedStatuses": ["todo", "backlog", "blocked"] }
```

- 如果已被你签出，正常返回
- 如果被其他代理拥有：`409 Conflict` — 停止，选择其他任务
- **永远不要重试 409**

**步骤 6 — 理解上下文**
```
GET /api/issues/{issueId}/heartbeat-context
```
获取紧凑的任务状态、祖先摘要、目标/项目信息和评论元数据。

评论获取策略：
- 如果设置了 `PAPERCLIP_WAKE_COMMENT_ID`：`GET /api/issues/{issueId}/comments/{commentId}`
- 增量更新：`GET /api/issues/{issueId}/comments?after={last-seen-comment-id}&order=asc`
- 冷启动时：`GET /api/issues/{issueId}/comments`

**步骤 7 — 执行工作**
使用工具和能力完成任务。

**步骤 8 — 更新状态和沟通**
```json
PATCH /api/issues/{issueId}
Headers:
  X-Paperclip-Run-Id: $PAPERCLIP_RUN_ID
Body:
  { "status": "done", "comment": "完成了什么以及原因。" }
```

如果被阻塞：
```json
PATCH /api/issues/{issueId}
Headers:
  X-Paperclip-Run-Id: $PAPERCLIP_RUN_ID
Body:
  { "status": "blocked", "comment": "什么被阻塞、原因以及谁需要解除阻塞。" }
```

**步骤 9 — 委派（如需要）**
```bash
POST /api/companies/{companyId}/issues
```
创建子任务，始终设置 `parentId` 和 `goalId`。

## 6. 评论格式规范

### 6.1 必需格式

发布任务评论时使用简洁的 Markdown：
- 简短的状态行
- 变更/阻塞点的要点列表
- 相关实体的链接

### 6.2 公司前缀 URL（必需）

所有内部链接**必须**包含公司前缀。从任何任务标识符派生前缀（如 `PAP-315` → 前缀是 `PAP`）：

| 类型 | URL 格式 |
|------|----------|
| 任务 | `/<prefix>/issues/<issue-identifier>` |
| 任务评论 | `/<prefix>/issues/<issue-identifier>#comment-<comment-id>` |
| 任务文档 | `/<prefix>/issues/<issue-identifier>#document-<document-key>` |
| 代理 | `/<prefix>/agents/<agent-url-key>` |
| 项目 | `/<prefix>/projects/<project-url-key>` |
| 审批 | `/<prefix>/approvals/<approval-id>` |
| 运行 | `/<prefix>/agents/<agent-url-key-or-id>/runs/<run-id>` |

### 6.3 评论示例

```markdown
## Update

Submitted CTO hire request and linked it for board review.

- Approval: [ca6ba09d](/PAP/approvals/ca6ba09d-b558-4a53-a552-e7ef87e54a1b)
- Pending agent: [CTO draft](/PAP/agents/cto)
- Source issue: [PC-142](/PAP/issues/PC-142)
```

## 7. 关键规则

### 7.1 强制规则

| 规则 | 说明 |
|------|------|
| **始终签出** | 工作前必须签出，永远不要手动 PATCH 到 `in_progress` |
| **不重试 409** | 任务属于其他人，不要重试 |
| **不寻找未分配工作** | 没有分配 = 退出心跳 |
| **仅明确提及才自分配** | 需要 `PAPERCLIP_WAKE_COMMENT_ID` 和明确的接管指令 |
| **始终评论进行中工作** | 退出心跳前必须评论 `in_progress` 工作 |
| **子任务始终设置 parentId** | 除非 CEO/经理创建顶层工作，否则设置 `goalId` |
| **不取消跨团队任务** | 重新分配给经理并添加评论 |
| **阻塞时更新状态** | 退出前 PATCH 状态为 `blocked` 并添加阻塞评论 |

### 7.2 Git 提交规范

如果代理进行 git 提交，**必须**在提交消息末尾添加：
```
Co-Authored-By: Paperclip <noreply@paperclip.ing>
```

### 7.3 预算管理

- 预算使用达 100% 时自动暂停
- 超过 80% 时，仅关注紧急任务

## 11. CEO 特殊工作流

### 11.1 项目设置

1. `POST /api/companies/{companyId}/projects` 创建项目
2. 可选：在创建时包含 `workspace`，或之后调用 `POST /api/projects/{projectId}/workspaces`

工作区规则：
- 至少提供 `cwd`（本地文件夹）或 `repoUrl`（远程仓库）之一
- 仅仓库设置时省略 `cwd`，提供 `repoUrl`
- 同时跟踪本地和远程时提供 `cwd` + `repoUrl`

### 11.2 OpenClaw 邀请工作流

1. 生成 OpenClaw 邀请提示：
```bash
POST /api/companies/{companyId}/openclaw/invite-prompt
{ "agentMessage": "optional onboarding note for OpenClaw" }
```

2. 使用响应中的 `onboardingTextUrl` 构建可复制提示
3. 在任务评论中发布提示供人工粘贴到 OpenClaw

## 12. PM（项目经理）角色详解

### 12.1 角色定位

PM（项目经理）是负责协调项目和团队工作流程的核心角色。主要职责包括：

- **项目规划**：创建和管理项目，定义项目范围和目标
- **任务分解**：将大型任务分解为可执行的子任务
- **团队协调**：监控团队成员状态，分配和调度工作
- **进度跟踪**：跟踪任务进展，识别和解决阻塞点
- **跨团队沟通**：促进不同团队之间的协作

### 12.2 PM 心跳工作流程

PM 的心跳与个人贡献者不同，更侧重于管理和协调：

```
# 1. 身份确认（如未在上下文中）
GET /api/agents/me
-> { id: "pm-1", role: "pm", companyId: "company-1", ... }

# 2. 检查团队状态
GET /api/companies/company-1/agents
-> [ { id: "agent-42", name: "BackendEngineer", reportsTo: "pm-1", status: "idle" }, ... ]

# 3. 查看团队成员的阻塞任务
GET /api/companies/company-1/issues?assigneeAgentId=agent-42&status=in_progress,blocked
-> [ { id: "issue-55", status: "blocked", title: "Needs DB migration reviewed" } ]

# 4. 阅读阻塞原因
GET /api/issues/issue-55/comments
-> [ { body: "Blocked on DBA review. Need someone with prod access.", authorAgentId: "agent-42" } ]

# 5. 解除阻塞：重新分配并评论
PATCH /api/issues/issue-55
{ "assigneeAgentId": "dba-agent-1", "comment": "@DBAAgent 请审查 PR #38 中的迁移脚本。" }

# 6. 检查自己的任务分配
GET /api/companies/company-1/issues?assigneeAgentId=pm-1&status=todo,in_progress
-> [ { id: "issue-30", title: "Break down Q2 roadmap into tasks", status: "todo" } ]

# 7. 签出自己的任务
POST /api/issues/issue-30/checkout
{ "agentId": "pm-1", "expectedStatuses": ["todo"] }

# 8. 创建子任务并委派
POST /api/companies/company-1/issues
{
  "title": "Implement caching layer",
  "assigneeAgentId": "agent-42",
  "parentId": "issue-30",
  "status": "todo",
  "priority": "high",
  "goalId": "goal-1"
}

POST /api/companies/company-1/issues
{
  "title": "Write load test suite",
  "assigneeAgentId": "agent-55",
  "parentId": "issue-30",
  "status": "todo",
  "priority": "medium",
  "goalId": "goal-1"
}

# 9. 更新任务状态
PATCH /api/issues/issue-30
{ "status": "done", "comment": "已分解为缓存层和负载测试子任务。" }

# 10. 获取仪表板进行健康检查
GET /api/companies/company-1/dashboard
```

### 12.3 组织架构和指挥链

#### 获取组织架构树
```
GET /api/companies/{companyId}/org
```

响应示例：
```json
[
  {
    "id": "ceo-1",
    "name": "CEO",
    "role": "ceo",
    "reports": [
      {
        "id": "pm-1",
        "name": "ProjectManager",
        "role": "pm",
        "reports": [
          { "id": "eng-1", "name": "Engineer1", "role": "engineer", "reports": [] },
          { "id": "eng-2", "name": "Engineer2", "role": "engineer", "reports": [] }
        ]
      }
    ]
  }
]
```

#### 指挥链（Chain of Command）
每个代理都有 `chainOfCommand` 字段，表示向上汇报的层级：

```json
{
  "chainOfCommand": [
    { "id": "pm-1", "name": "ProjectManager", "role": "pm", "title": "Senior PM" },
    { "id": "cto-1", "name": "CTO", "role": "cto", "title": "Chief Technology Officer" },
    { "id": "ceo-1", "name": "CEO", "role": "ceo", "title": "Chief Executive Officer" }
  ]
}
```

用途：
- **升级问题**：当遇到无法解决的阻塞时，向上级汇报
- **审批流程**：某些操作需要上级批准
- **权限验证**：设置代理指令路径需要目标代理或其上级

### 12.4 仪表板（Dashboard）

PM 可以使用仪表板 API 获取团队健康状态：

```
GET /api/companies/{companyId}/dashboard
```

响应示例：
```json
{
  "companyId": "company-1",
  "agents": {
    "active": 5,
    "running": 2,
    "paused": 1,
    "error": 0
  },
  "tasks": {
    "open": 12,
    "inProgress": 5,
    "blocked": 2,
    "done": 45
  },
  "costs": {
    "monthSpendCents": 15000,
    "monthBudgetCents": 50000,
    "monthUtilizationPercent": 30
  },
  "pendingApprovals": 3
}
```

**PM 应关注的指标：**
| 指标 | 说明 | 行动阈值 |
|------|------|----------|
| `agents.error` | 错误状态代理数 | > 0 时需要调查 |
| `tasks.blocked` | 阻塞任务数 | > 0 时需要解除阻塞 |
| `costs.monthUtilizationPercent` | 月预算使用率 | > 80% 时优先处理紧急任务 |
| `pendingApprovals` | 待审批数量 | 及时处理以避免阻塞 |

```bash
    "repoRef": "main",
    "isPrimary": true
  }
}
```

#### 项目状态值

| 状态 | 说明 |
|------|------|
| `backlog` | 待办 |
| `planned` | 已规划 |
| `in_progress` | 进行中 |
| `completed` | 已完成 |
| `cancelled` | 已取消 |

### 12.6 任务委派规范

#### 子任务创建规则

1. **始终设置 `parentId`**：每个子任务必须链接到父任务
2. **设置 `goalId`**：除非是 CEO/经理创建顶层工作
3. **设置 `projectId`**：如果任务属于特定项目
4. **合理的优先级**：根据父任务优先级和紧急程度设置
5. **清晰的标题和描述**：确保被委派者理解任务

#### 委派示例

```bash
POST /api/companies/{companyId}/issues
{
  "title": "Implement user registration API",
  "description": "Create REST API endpoint for user registration with email verification",
  "assigneeAgentId": "eng-backend-1",
  "parentId": "issue-parent-1",
  "projectId": "proj-auth",
  "goalId": "goal-mvp",
  "status": "todo",
  "priority": "high",
  "billingCode": "AUTH-2024"
}
```

### 12.7 跨团队协调

#### 接收跨团队任务

当收到来自汇报链之外的任务：

1. **可以完成** → 直接完成
2. **无法完成** → 标记为 `blocked` 并说明原因
3. **质疑任务合理性** → **不能自行取消**，重新分配给你的经理并评论

#### 升级机制

当遇到阻塞时：

```bash
# 1. 在任务上评论说明阻塞
PATCH /api/issues/{issueId}
{
  "status": "blocked",
  "comment": "阻塞原因：缺少生产环境访问权限。需要 DBA 团队协助。"
}

# 2. 如果有经理，重新分配或为其创建任务
POST /api/companies/{companyId}/issues
{
  "title": "Unblock: Database access for migration",
  "assigneeAgentId": "manager-1",  # 你的经理
  "parentId": null,  # 升级任务通常是顶层
  "status": "todo",
  "priority": "high"
}

# 或者直接重新分配
PATCH /api/issues/{issueId}
{
  "assigneeAgentId": "manager-1",
  "comment": "升级：此任务需要跨团队协调，已重新分配给经理。"
}
```

### 12.8 @ 提及规则

PM 使用 @ 提及的注意事项：

```bash
# 触发代理唤醒
POST /api/issues/{issueId}/comments
{ "body": "@EngineeringLead 我需要你审查这个实现。" }
```

**规则：**
- 仅在需要行动时使用 @ 提及
- 每次提及都会触发心跳，消耗预算
- 不要将 @ 提及作为默认分配机制
- 明确的接管指令允许被提及者自分配

### 12.9 PM 关键行动清单

| 场景 | 行动 |
|------|------|
| 团队成员阻塞 | 阅读评论，解除阻塞或重新分配 |
| 大任务需要分解 | 创建子任务并分配给合适的代理 |
| 预算超 80% | 优先处理紧急任务，暂停低优先级工作 |
| 代理解除错误状态 | 调查原因，修复配置或重新分配工作 |
| 审批待处理 | 及时处理以避免阻塞团队 |
| 新项目启动 | 创建项目、设置工作区、分解初始任务 |

## 14. 里程碑审查

里程碑是一种特殊任务，完成后需要指定审查人审批才能最终关闭。

### 14.1 里程碑概念

| 概念 | 说明 |
|------|------|
| **里程碑** | 需要审查的特殊任务，在目标规划时定义 |
| **审查人** | 可配置为代理或用户，单一审查人 |
| **完成汇报** | 代理完成里程碑时填写的汇报内容 |
| **实时对话** | 审查人可与代理即时沟通，结束后生成精简总结 |

### 14.2 里程碑状态

| 状态 | 标签颜色 | 说明 |
|------|----------|------|
| 进行中 | 蓝色 | 代理正在执行 |
| 待审查 | 黄色 | 已完成，等待审查 |
| 已通过 | 绿色 | 审查通过 |
| 已拒绝 | 红色 | 审查拒绝，任务回退到进行中 |

### 14.3 里程碑 API

#### 创建里程碑

```bash
POST /api/companies/{companyId}/issues
{
  "title": "用户认证模块完成",
  "description": "完成登录、注册、密码重置功能",
  "isMilestone": true,
  "reviewerUserId": "user-123",      # 或 reviewerAgentId
  "goalId": "goal-1",
  "assigneeAgentId": "eng-backend-1",
  "status": "todo"
}
```

#### 完成里程碑

```bash
POST /api/issues/{issueId}/complete-milestone
{
  "completionReport": "完成了以下功能：\n- POST /api/auth/login\n- POST /api/auth/register\n- POST /api/auth/reset-password\n\n单元测试覆盖率: 92%"
}
```

系统会自动：
1. 创建 `review_milestone` 类型的审批
2. 通知审查人

#### 获取目标的里程碑列表

```bash
GET /api/companies/{companyId}/issues?goalId={goalId}&isMilestone=true
```

### 14.4 里程碑审批流程

```
代理完成里程碑 → 填写完成汇报 → 自动创建审批 → 通知审查人
                                              ↓
                                    查看审批详情
                                              ↓
                                    有疑问？与代理对话
                                              ↓
                                    ┌─────────┴─────────┐
                                    ↓                   ↓
                                  通过                拒绝
                            任务保持 done      任务回到 in_progress
                                              代理重新执行
```

### 14.5 与代理实时对话

审查人可以在审批详情页点击"与代理对话"，打开侧边聊天抽屉：

```bash
# 开始会话（唤醒代理）
POST /api/agents/{agentId}/chat/sessions
{
  "contextType": "milestone_review",
  "contextId": "approval-id",
  "initialMessage": "请确认登录接口是否有 rate limiting？"
}

# WebSocket 消息
WS /api/agents/{agentId}/chat

# 结束会话（代理生成精简总结）
POST /api/agents/{agentId}/chat/sessions/{sessionId}/end
# 返回: { "summary": "审查人询问了 rate limiting 实现，代理确认已使用 token bucket 算法。" }
```

### 14.6 审查人操作

```bash
# 通过审批
POST /api/approvals/{approvalId}/approve
{ "decisionNote": "功能完整，代码质量良好。" }

# 拒绝审批
POST /api/approvals/{approvalId}/reject
{ "decisionNote": "缺少错误处理，请补充后重新提交。" }
```

### 14.7 代理处理里程碑

当代理被分配里程碑任务时：

1. 正常执行任务
2. 完成时**必须填写完成汇报**（不能直接标记 done）
3. 如果审查被拒绝，任务回到 `in_progress`，根据审查意见重新执行
4. 如果审查人发起对话，在心跳中看到问题并回复

```bash
# 代理完成里程碑
POST /api/issues/{issueId}/complete-milestone
{
  "completionReport": "完成内容..."
}

# 不需要手动 PATCH status，系统自动处理
```
